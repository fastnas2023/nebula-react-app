# Web 应用 UX/UI 全面测试避坑手册 (完整版)

这份手册总结了主流视频会议应用及现代 Web 产品中容易出现的 UX/UI 交互问题。它是前端开发、UI 设计师和 QA 团队在交付前必须对照检查的 Checklist。

---

## 📱 一、 移动端专属体验陷阱 (Mobile Web Specifics)

移动端不是把桌面端变窄那么简单，它有完全不同的物理交互和系统环境。

*   **[ ] 软键盘遮挡与弹起 (Keyboard Obscuration)**
    *   **现象**：用户点击底部的输入框，系统软键盘弹起，直接把输入框或者“提交”按钮给盖住了，用户根本看不见自己敲了什么。
    *   **验收标准**：在真实手机上测试。点击表单最底部的输入框，确保页面能自动滚动（`scrollIntoView`），让输入框和提交按钮始终保持在键盘上方。不要在包含输入框的页面使用绝对定位在底部的按钮（`bottom: 0`）。
*   **[ ] 100vh 滚动条跳动 (The 100vh Bug)**
    *   **现象**：开发者把首屏高度设为 `100vh`。但在 Safari 或 Chrome 移动版中，由于顶/底部地址栏会在滚动时自动收缩/展开，导致 `100vh` 的实际像素值不断变化，页面出现剧烈跳动。
    *   **验收标准**：永远不要在移动端使用 `100vh`。请使用现代 CSS 视口单位：`100dvh` (Dynamic Viewport Height) 或 `100svh` (Small Viewport Height)。
*   **[ ] 刘海屏与底部小黑条 (Safe Area Insets)**
    *   **现象**：把返回按钮做到了屏幕绝对的左上角，结果被 iPhone 的“刘海”或“灵动岛”挡住根本点不到；或者把底部导航栏贴紧屏幕底边缘，结果被 iOS 的“Home 键小黑条”遮挡，导致极易误触退回桌面。
    *   **验收标准**：必须使用 CSS 环境变量预留安全距离：`padding-top: env(safe-area-inset-top)` 和 `padding-bottom: env(safe-area-inset-bottom)`。
*   **[ ] 幽灵 Hover 与移动端悬停缺失 (No Hover on Touch)**
    *   **现象**：PC 端把重要的“删除”按钮隐藏在 Hover 状态下。但在手机上根本没有 Hover，导致用户永远找不到删除入口；或者点击某个卡片时，第一次点击只触发了 Hover 样式，第二次点击才触发跳转，体验极其怪异。
    *   **验收标准**：**绝对不要把核心操作或关键信息仅仅绑定在 `:hover` 状态下**。移动端必须默认展示所有关键操作图标，或者提供“左滑/长按”等触控专属的替代方案。对于移动端不需要 Hover 的元素，可通过 `@media (hover: hover)` 媒体查询来隔离 Hover CSS。
*   **[ ] 边缘滑动手势冲突 (Gesture Conflicts)**
    *   **现象**：在页面左侧做了一个横向滑动的轮播图或抽屉，用户在手机上向右滑想看上一张图，结果触发了 iOS 系统的“边缘右滑返回上一页”，直接退出了网站。
    *   **验收标准**：避免在屏幕绝对边缘（尤其是左边缘的 20px 内）放置需要重度横向滑动的组件。如果必须滑动，请使用 `touch-action: pan-y` 显式声明，防止纵向滚动干扰。
*   **[ ] 唤起错误的数字键盘 (Wrong Input Mode)**
    *   **现象**：让用户输入手机验证码或会议 PIN 码，结果弹出的却是全键盘（带字母），用户还要自己手动切到数字键盘，极大地增加了输入阻力。
    *   **验收标准**：对于纯数字输入，必须设置 `inputmode="numeric"`（有时配合 `pattern="[0-9]*"`）。这会直接唤起又大又好用的九宫格数字键盘。
*   **[ ] 长按选中与系统菜单干扰 (Text Selection Disable)**
    *   **现象**：用户在疯狂点击“加入会议”按钮时，不小心点长了，结果弹出了系统的“拷贝/查询/分享”菜单，或者把整个按钮的文字高亮选中了。
    *   **验收标准**：对于所有的按钮、图标、导航栏等纯交互元素（非阅读类文本），必须加上 CSS `user-select: none` 和 `-webkit-touch-callout: none`，屏蔽系统长按菜单。

## 🖱️ 二、 鼠标、悬浮与导航交互 (Navigation & Pointer)

*   **[ ] 盲盒式导航 / 神秘肉导航 (Mystery Meat Navigation)**
    *   **现象**：侧边栏折叠后，只剩下一排纯图形图标（Icon-only），且没有任何文字提示。用户在点击前完全不知道这个图标会跳转到哪里，强迫用户记忆，认知负荷极高。
    *   **验收标准**：对于非全球通用的图标（如除了“放大镜=搜索”之外的所有图标），如果空间受限必须折叠，**必须提供明确的 Tooltip（工具提示）**。当鼠标悬停时，立刻显示图标对应的文字标签。
*   **[ ] 强制折叠与用户控制权丧失 (Forced Collapse & Loss of Control)**
    *   **现象**：在较窄的屏幕上，系统强制把侧边栏折叠起来，且不提供任何展开的按钮。如果右侧主内容区其实有足够空间，这种“一刀切”的响应式设计会剥夺用户的控制权，导致用户无法快速浏览完整菜单。
    *   **验收标准**：响应式折叠应该是默认行为，而不是强制行为。必须提供一个显式的切换按钮（如面板展开/收起图标），允许用户在窄屏下手动展开侧边栏（Progressive Disclosure），将控制权交还给用户。
*   **[ ] 修复悬浮间隙 (Hover Gap / Diagonal Tracking)**
    *   **现象**：鼠标从触发元素移动到弹出的下拉菜单/Tooltip 途中，因为物理缝隙导致菜单瞬间消失。
    *   **验收标准**：用透明的 `padding` 填补触发器和菜单之间的物理间隙。
*   **[ ] 消除幽灵悬浮 (Ghost Hover)**
    *   **现象**：点击某个按钮打开了弹窗或跳转了页面，当弹窗关闭/返回时，原先底部的按钮依然保持着 Hover 的高亮残影状态。
    *   **验收标准**：交互完成后（如弹窗关闭），触发元素的样式应完全恢复到 Default 状态。
*   **[ ] 区分键盘与鼠标焦点 (Focus Outline)**
    *   **现象**：用户用鼠标点击了一个按钮，浏览器默认的粗边框（Outline）一直残留，严重破坏 UI 美感。
    *   **验收标准**：使用 `:focus-visible` 替代全局的 `:focus`。确保用键盘 `Tab` 键导航时有清晰的焦点框，而用鼠标点击时不出现。

## 🎯 三、 状态可见性与触控目标 (State Visibility & Click Targets)

*   **[ ] 状态指代不明与视觉形变 (State Ambiguity & Visual Distortion)**
    *   **现象**：
        1. 比如“切换开关 (Toggle Switch)”，激活和未激活状态没有明显的色彩或滑块位移对比，用户分不清到底是开还是关。
        2. UI 元素（如开关背景或圆形按钮）在不同尺寸下被强行拉伸或挤压，变成了难看的椭圆或胶囊形（Squished UI elements）。
    *   **验收标准**：
        1. **状态必须有双重反馈**：开启状态不仅要有颜色变化（如从灰变蓝），还要有明确的几何位移（滑块从左到右）。
        2. **锁定形变**：给固定比例的 UI 元素设置硬性的宽高（如 `w-12 h-6`）和圆角，并使用绝对定位避免被外层 Flex/Grid 布局挤压变形。
*   **[ ] 确保充足的触控目标 (Fitts's Law)**
    *   **现象**：图标或文字看起来很大，但实际可点击的标签区域极小，用户稍微点偏就无反应。
    *   **验收标准**：移动端确保所有独立交互元素的触控区域至少达到 `44x44px`（可通过增加 padding 实现，而非放大图标本身）。
*   **[ ] 必须存在的加载状态 (Missing Loading State)**
    *   **现象**：点击“提交”后网络延迟导致界面毫无反应，用户狂点多次，引发重复提交。
    *   **验收标准**：在 Network 面板开启 `Slow 3G` 测试。点击后必须立刻看到 Spinner，且按钮变为 `disabled` 状态。
*   **[ ] 防止模态框点穿与滚动溢出 (Scroll Bleed)**
    *   **现象**：打开了一个弹窗，但滚动时背后的底层页面居然跟着一起滚动了。
    *   **验收标准**：打开弹窗，疯狂滚动。确保底层的 `<body>` 被锁死了滚动（通过动态添加 `overflow: hidden`）。

## 📝 四、 表单与输入体验 (Forms & Inputs)

*   **[ ] 杜绝毁灭性的表单清空**
    *   **现象**：用户填了长长的表单，因为某一项后端校验失败，页面刷新导致用户输入的所有内容全被清空。
    *   **验收标准**：后端报错打回时，表单必须保留用户之前输入的所有合法内容。
*   **[ ] 拒绝严苛的格式强迫症 (Forgiving Inputs)**
    *   **现象**：逼迫用户必须按照特定格式输入（例如日期必须用 `/` 分隔，或者手机号必须加 `-`）。
    *   **验收标准**：让机器去适应人。允许用户输入冗余字符，前端在 `onBlur` 时自动过滤、格式化和解析。
*   **[ ] 把握表单报错的最佳时机 (Validation Timing)**
    *   **现象**：用户刚敲了第一个字母，输入框立刻爆红提示“格式不正确”，带来极强的挫败感。
    *   **验收标准**：离开输入框时 (`onBlur`) 进行首次报错；报错出现后，用户修改文字时 (`onChange`) 实时消除报错提示。

## 👁️ 五、 视觉响应与布局 (Visual & Layout)

*   **[ ] 消除布局抖动 (Cumulative Layout Shift - CLS)**
    *   **现象**：页面加载到一半，上面突然加载出一张大图把按钮挤下去，导致用户点错了地方。
    *   **验收标准**：节流网络（Slow 3G）观察页面。给所有图片、视频容器预设明确的宽高比（`aspect-ratio`），预留占位空间。
*   **[ ] 品牌一致性与动效丢失 (Brand Consistency & Missing Animations)**
    *   **现象**：在不同的页面或组件状态（如侧边栏折叠/展开）切换时，原本精心设计的品牌 Logo 动效突然消失，变成了静态图片或普通文字，破坏了整体的精致感。
    *   **验收标准**：核心品牌元素必须封装为独立的组件（如 `NebulaLogo`）并在各处复用。进行布局调整时，务必保留其原有的交互动效（Hover 态、Pulse 等），确保全站视觉体验高度统一。
*   **[ ] 测试极端数据的“破坏力” (Edge Text Cases)**
    *   **现象**：真实用户的名字可能叫 "Hubert Blaine Wolfeschlegelsteinhausenbergerdorff"，直接把布局撑爆或重叠。
    *   **验收标准**：用超长无空格文本去测试每一个展示字段。确保配置了 `truncate` 或 `break-words`。

## ⚡ 六、 性能错觉与特定场景 (Performance & Scenarios)

*   **[ ] 避免悲观的 UI 交互 (Pessimistic UI)**
    *   **现象**：点赞或静音后，前端非要等服务器 2 秒钟返回成功后才改变图标状态，系统显得极其卡顿。
    *   **验收标准**：使用乐观 UI (Optimistic UI)。用户一点击，前端立刻改变视觉状态。如果后台报错，再把状态回滚并弹出轻提示。
*   **[ ] 拯救丑陋的“空状态” (Dead-end Empty States)**
    *   **现象**：用户第一次进来没有任何记录，页面中间只有一个干巴巴的文字：“暂无数据”。
    *   **验收标准**：空状态必须包含三要素：1. 解释为什么为空；2. 友好的插图；3. 明确的行动号召 (CTA 主按钮)。

## 🛑 七、 严禁使用的“暗黑模式” (Dark Patterns)

*   **[ ] 捕蟑螂匣 (Roach Motel)**
    *   **现象**：注册/订阅只需 1 秒，但想注销账号或取消订阅，却需要找人工客服、发邮件、等 7 天。
    *   **规范**：高级的 UX 应该让离开和加入一样简单。提供明确的一键取消/退出入口。
*   **[ ] 隐性视觉误导 (Misdirection)**
    *   **现象**：把“取消”做成高亮的红色主按钮，把真正的“确认”做成灰色不可点的次要文本，故意误导用户点错。
    *   **规范**：遵循常规视觉隐喻：主干操作高亮，破坏性操作标红或二次确认，取消操作低调。
